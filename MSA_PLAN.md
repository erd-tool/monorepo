# ERD Tool MSA 전환 계획

- 작성 기준일: 2026-04-03
- 목적: 현재 단일 백엔드 구조를 MSA 환경으로 전환할 때 참조할 기준 문서
- 전환 원칙: 한 번에 전부 쪼개지 않고, 업무 경계를 기준으로 점진적으로 분리

## 1. 전환 목표

- 공통 인증을 Gateway에서 일원화
- 서비스별 책임과 데이터 소유권 분리
- 동기 호출과 비동기 이벤트를 역할에 맞게 분리
- 로컬 Docker Compose 환경에서도 서비스 간 구조를 재현 가능하게 구성
- 향후 Kubernetes 또는 클라우드 배포 구조로 확장 가능한 형태 확보

## 2. 목표 아키텍처 개요

### 2.1 권장 서비스 구성

- `gateway-service`
  - Spring Cloud Gateway
  - 외부 진입점
  - JWT 검증
  - 사용자 식별 헤더 주입
  - 라우팅, CORS, 요청 추적

- `auth-service`
  - 회원가입, 로그인, 내 정보 조회
  - JWT 발급
  - 사용자 계정 관리

- `team-service`
  - 팀 생성/조회
  - 팀원 초대
  - 초대 수락
  - 팀 멤버십 및 역할 관리

- `erd-service`
  - ERD 생성/조회/수정/삭제
  - 문서 메타데이터와 본문 관리
  - SQL export

- `collaboration-service`
  - Yjs WebSocket 협업 서버
  - 세션/room 관리
  - 필요 시 브로커와 연동해 다중 인스턴스 확장

- 선택적 후속 서비스
  - `notification-service`
  - `audit-service`
  - `file-export-service`

### 2.2 요청 흐름

1. 클라이언트는 모든 HTTP 요청을 `gateway-service`로 보낸다.
2. Gateway는 JWT를 검증한다.
3. Gateway는 내부 서비스에 사용자 식별 헤더와 추적 헤더를 주입한다.
4. 내부 서비스는 Gateway가 전달한 인증 컨텍스트를 기준으로 비즈니스 로직을 수행한다.
5. 필요한 경우 서비스 간 동기 호출은 REST + `FeignClient`, 비동기 전달은 Broker를 이용한다.

## 3. 핵심 설계 원칙

### 3.1 서비스 경계는 업무 기준으로 나눈다

- 인증은 `auth-service`
- 팀과 권한은 `team-service`
- ERD 문서는 `erd-service`
- 실시간 협업은 `collaboration-service`

### 3.2 Gateway를 신뢰의 시작점으로 둔다

- 외부 요청은 반드시 Gateway를 통과
- 내부 서비스는 클라이언트가 직접 보낸 `X-USER-*` 헤더를 신뢰하지 않음
- 가능하면 내부 네트워크 분리 또는 헤더 서명 전략을 함께 사용

### 3.3 동기/비동기 통신을 분리한다

- 동기 통신
  - 사용 예: 사용자 정보 조회, 팀 권한 확인, 즉시 검증
  - 방식: REST + `FeignClient`

- 비동기 통신
  - 사용 예: 팀 초대 이벤트, ERD 생성 이벤트, 감사 로그, 알림 발송
  - 방식: Kafka 또는 RabbitMQ

### 3.4 데이터 소유권을 분리한다

- 서비스는 자신의 데이터만 직접 관리
- 타 서비스 DB 직접 조회 금지
- 초기 단계에서는 DB는 하나를 쓰더라도 최소한 스키마 분리를 권장
- 장기적으로는 서비스별 DB 분리를 목표로 한다

## 4. 권장 인증 구조

### 4.1 추천 처리 방식

- JWT 발급은 `auth-service`
- JWT 검증은 `gateway-service`
- Gateway는 아래 정보를 내부 헤더로 전달
  - `X-USER-ID`
  - `X-USER-LOGIN-ID`
  - `X-USER-ROLE`
  - `X-REQUEST-ID`

### 4.2 주의 사항

- `X-USER-ID` 하나만 전달하면 권한 분기 정보가 부족할 수 있음
- 내부 서비스를 외부에 직접 노출하면 헤더 위조 위험이 생김
- 따라서 "클라이언트 -> Gateway -> 내부 서비스" 경로를 강제하는 것이 중요함

## 5. FeignClient 사용 기준

### 적합한 경우

- 팀 소속 여부 조회
- 사용자 기본 정보 조회
- 즉시 응답이 필요한 동기 검증

### 지양할 경우

- 단순 이벤트 알림
- 후처리 작업
- 서비스 간 대량 브로드캐스트

### 예시

- `erd-service`가 팀 문서 생성 전에 `team-service`에 멤버십 검증 요청: 적합
- `team-service`가 초대 생성 후 알림 전송을 다른 서비스에 요청: Broker가 더 적합

## 6. Broker 사용 기준

### 6.1 Kafka를 선택하기 좋은 경우

- 이벤트 스트림 보관이 중요할 때
- 추후 분석/재처리 가능성이 클 때
- 서비스 수가 늘어나고 구독자가 많아질 때

### 6.2 RabbitMQ를 선택하기 좋은 경우

- 비교적 단순한 큐 기반 비동기 처리
- 학습 및 초기 운영 난이도를 낮추고 싶을 때
- 작업 분배와 소비자 제어가 중요한 경우

### 6.3 현재 프로젝트 기준 추천

현재 단계에서는 RabbitMQ가 더 단순하고 빠르게 도입 가능하다.  
이벤트 재처리, 스트림 기반 분석, 다수 소비자 패턴까지 염두에 둔다면 Kafka도 가능하지만 학습/운영 부담은 더 크다.

## 7. Docker Compose 환경 원칙

- 로컬 개발 환경에서는 서비스 간 호출 호스트로 `docker-compose` 서비스명을 사용한다.
- 예:
  - `http://auth-service:8080`
  - `http://team-service:8080`
  - `http://erd-service:8080`
- 이 규칙은 로컬/도커 환경 기준이며, 장기적으로는 환경별 설정 분리가 필요하다.

## 8. 단계별 전환 플랜

### Phase 1. 사전 정리

- 현재 단일 백엔드의 패키지 경계를 명확히 정리
- 인증, 팀, ERD 모듈 간 의존성 파악
- 공통 DTO/예외/응답 구조 정리
- 내부 API 계약 후보 도출

### Phase 2. Gateway 도입

- `gateway-service` 생성
- 인증이 필요한 경로와 공개 경로 분리
- JWT 검증 필터 적용
- 내부 전달 헤더 규약 정의
- 프론트의 API 진입점을 Gateway로 변경

### Phase 3. auth-service 분리

- 사용자 계정과 로그인 기능 분리
- JWT 발급 책임 이전
- 기존 인증 관련 로직 제거 또는 축소

### Phase 4. team-service 분리

- 팀, 팀원, 초대 도메인 분리
- ERD 생성 전 팀 권한 확인을 `FeignClient`로 연결

### Phase 5. erd-service 분리

- ERD 문서 CRUD와 SQL export 책임 분리
- 팀 문서 접근 시 `team-service`와 권한 연동

### Phase 6. Broker 도입

- 초대 생성, ERD 생성/수정, 감사 로그 등의 이벤트 정의
- RabbitMQ 또는 Kafka 연결
- 후처리/알림 흐름 비동기화

### Phase 7. collaboration-service 연계 고도화

- 협업 서버의 인증 연계 방식 정리
- 필요 시 Broker 또는 Redis 기반 다중 인스턴스 확장 검토

## 9. 현재 프로젝트에 맞는 우선순위

### 1순위

- Gateway 도입
- 인증 헤더 규약 확정
- 프론트 API 진입점 단일화

### 2순위

- `auth-service`, `team-service`, `erd-service` 경계 확정
- 내부 API 계약서 초안 작성

### 3순위

- `team-service`와 `erd-service` 분리
- Feign 기반 권한 조회 연결

### 4순위

- Broker 도입
- 초대/알림/감사 로그 이벤트 비동기화

## 10. 예상 리스크

- 서비스 분리 후 트랜잭션 경계가 끊기면서 데이터 일관성 처리 난이도 증가
- 단일 서비스보다 로컬 실행과 디버깅 복잡도 상승
- 네트워크 호출 증가로 장애 지점 확대
- 브로커 도입 시 재시도, 중복 소비, 장애 복구 전략 필요
- 수업용 데모 환경에서는 운영 복잡도가 빠르게 커질 수 있음

## 11. 권장 산출물

MSA 전환 전에 아래 문서를 먼저 만드는 것을 권장한다.

- 서비스 분리 다이어그램
- 내부 API 계약서
- 이벤트 목록 정의서
- 인증 헤더 규약 문서
- Docker Compose 서비스 토폴로지 문서

## 12. 한 줄 가이드

이 프로젝트의 MSA 전환은 "Feign과 Broker를 모두 쓴다"가 핵심이 아니라,  
"업무 경계를 명확히 나누고, Gateway를 신뢰의 시작점으로 세우며, 동기/비동기 통신을 역할에 맞게 분리한다"가 핵심이다.
