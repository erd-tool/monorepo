# ERD Collaboration Tool 프로젝트 정리

- 작성 기준일: 2026-04-03
- 작성 기준: 현재 저장소 코드 구조와 구현 상태를 기준으로 정리
- 목적: 개발 플랜, 요구 사항, 구현된 내용, 진행 상황을 한 문서에서 빠르게 파악하기 위함

## 1. 프로젝트 개요

이 프로젝트는 웹 기반 협업 ERD 도구를 만드는 것을 목표로 한다.  
사용자는 로그인 후 개인 또는 팀 단위로 ERD 문서를 생성하고, 엔티티와 관계를 시각적으로 편집하며, 실시간 협업과 SQL/PNG 내보내기를 사용할 수 있다.

현재 구조는 다음과 같이 구성되어 있다.

- `backend`: Spring Boot 기반 인증, 팀, ERD 문서 관리 API
- `frontend`: React + React Flow 기반 ERD 편집 UI
- `collaboration`: Yjs WebSocket 기반 실시간 동기화 서버
- `docker-compose.yml`: PostgreSQL, Redis, Backend, Frontend, Collaboration 통합 실행

## 2. 개발 목표와 MVP 플랜

### 2.1 최종 목표

- 회원 인증과 권한 제어가 가능한 협업형 ERD SaaS 형태의 기반 구축
- 팀 단위 ERD 작성 및 공동 편집 지원
- ERD 시각 편집 결과를 SQL 스키마로 변환
- PNG 등 산출물 export 제공
- Docker 기반 로컬 통합 실행 환경 제공

### 2.2 단계별 개발 플랜

#### Phase 1. 서비스 기반 구축

- 회원가입 / 로그인 / 내 정보 조회
- JWT 인증 및 보호된 API 구성
- PostgreSQL 연동 및 기본 도메인 테이블 구성

#### Phase 2. 협업 단위 구성

- 팀 생성
- 팀 목록 조회
- 팀 초대 토큰 발급
- 초대 수락 처리

#### Phase 3. ERD 문서 관리

- ERD 생성 / 조회 / 수정 / 삭제
- 개인 ERD와 팀 ERD 구분
- 문서 제목, 설명, 공개 여부 관리

#### Phase 4. 시각 편집기

- 엔티티 추가 / 수정 / 삭제
- 필드 추가 / 수정 / 삭제
- 관계 생성 / 수정 / 삭제
- 드래그 기반 위치 이동
- Undo / Redo
- 자동 저장

#### Phase 5. 협업 및 산출물

- Yjs 기반 실시간 동기화
- 접속자 상태 표시
- SQL export
- PNG export

#### Phase 6. 고도화

- 메모 노트 기능 UI 완성
- 공개 문서 공유 UX
- 초대 수락 화면 및 팀 운영 기능 보강
- SQL export 정합성 강화
- Redis 활용 고도화 및 운영 안정화

## 4. 현재 구현된 내용

### 4.1 백엔드 구현

#### 인증/보안

- `/api/auth/signup`, `/api/auth/login`, `/api/auth/me` 제공
- Spring Security + JWT 필터 기반 인증 처리
- BCrypt 비밀번호 암호화 적용
- 인증 API와 `GET /healthz`를 제외한 나머지 API는 인증 필요

#### 팀 기능

- 팀 생성 API 제공
- 로그인 사용자의 팀 목록 조회 API 제공
- OWNER 권한 사용자만 팀원 초대 가능
- 이메일 기반 초대 토큰 발급
- 초대 토큰 수락 API 제공

#### ERD 문서 기능

- ERD 생성, 목록 조회, 상세 조회, 수정, 삭제 API 제공
- 개인 문서와 팀 문서 구분 저장
- 팀 소속 사용자에 대한 문서 접근 제어
- 문서 메타데이터: 제목, 설명, 공개 여부 저장
- 문서 본문은 `contentJson`으로 저장
- SQL export API 제공

### 4.2 프론트엔드 구현

#### 인증 및 대시보드

- 로그인 / 회원가입 화면 구현
- 백엔드 미연결 시 로컬 데모 세션으로 진입 가능
- 대시보드에서 팀 생성, ERD 생성, 최근 ERD 목록 조회 가능
- 팀 초대 토큰 발급 UI 제공

#### ERD 편집기

- React Flow 기반 캔버스 편집기 구현
- 엔티티 추가, 이름 변경, 색상 변경, 설명 입력 가능
- 필드 추가/삭제 및 속성 편집 가능
- 관계 생성 모드 제공
- 관계의 식별/비식별, 카디널리티, 참여성 설정 가능
- 관계선 선택 및 제어점 이동 가능
- 논리명 / 물리명 / 논리+물리 뷰 모드 제공
- 인라인 제목 수정, 설정 모달, 공개 여부 변경 가능
- Undo / Redo 단축키와 버튼 제공
- 자동 저장 처리
- PNG export 제공
- SQL 미리보기 및 다운로드 제공

#### 상태 관리

- Zustand persist를 이용한 로컬 상태 저장
- 문서별 workspace, undo/redo stack 관리
- 백엔드 실패 시 로컬 데이터 기반 동작 가능

### 4.3 실시간 협업 서버 구현

- 별도 Node.js 서버에서 Yjs WebSocket 연결 처리
- 문서 ID 기반 room 연결
- 접속 상태 및 접속자 awareness 반영
- 헬스체크 엔드포인트 제공

### 4.4 인프라 구성

- Dockerfile 분리 구성
- `docker-compose.yml`로 DB, 캐시, API, 협업 서버, 프론트 통합 실행 가능
- 환경 변수 기반 포트/DB/JWT 설정 가능

### 종합 판단

현재 프로젝트는 "협업 ERD 도구 MVP 1차 버전" 수준까지는 상당 부분 구현되어 있다.  
특히 인증, 팀, ERD CRUD, 시각 편집기, 자동 저장, 실시간 동기화, PNG/SQL 산출물까지 핵심 흐름이 이어져 있어 데모와 수업용 시연에는 충분히 활용 가능하다.

다만 상용 수준 또는 팀 실사용 수준으로 가기 위해서는 초대 수락 UX, 공개 공유 정책, SQL export 정합성, 메모 기능 연결, 운영 안정화가 추가로 필요하다.

## 6. 확인된 보완 포인트

### 6.1 메모 노트 기능은 부분 구현 상태

- 상태 저장소에는 `notes` 추가/수정/삭제 로직이 있음
- `NoteNode` 컴포넌트 파일도 존재함
- 그러나 현재 편집기 화면에서는 노트 노드가 실제 캔버스에 연결되어 있지 않아 사용자 기능으로는 완성되지 않음

### 6.2 팀 초대 수락 UX가 부족함

- 백엔드에는 초대 수락 API가 존재함
- 프론트 대시보드에서는 초대 토큰 발급까지만 보이며, 토큰 입력 후 수락하는 사용자 흐름 화면은 없음

### 6.3 공개 ERD는 메타데이터 수준에 가까움

- 문서에는 `public/private` 속성이 존재함
- 하지만 비로그인 사용자가 공개 문서를 조회하는 공개 URL, 공유 페이지, 읽기 전용 화면은 아직 구현되어 있지 않음

### 6.4 SQL export 정합성 보완 필요

- 프론트 문서 모델은 필드 타입 속성으로 `type`을 사용
- 백엔드 SQL export 서비스는 `dataType` 키를 기준으로 SQL을 생성
- 현재 구조에서는 백엔드 export 결과가 프론트 문서 구조와 완전히 일치하지 않을 수 있음
- 프론트에서 별도 DDL 생성 fallback을 가지고 있어 사용자 경험은 유지되지만, 서버 기준 SQL 생성 로직은 정리 필요

### 6.5 협업 서버의 Redis 활용은 아직 제한적

- 인프라에는 Redis가 포함되어 있음
- 현재 협업 서버 코드에서는 Redis를 직접 사용하는 로직이 보이지 않음
- 향후 세션 확장, 문서 브로드캐스트 최적화, 멀티 인스턴스 대응 시 활용 가능

### 6.6 팀 문서 권한 정책은 아직 단순함

- 현재는 팀 소속 여부 중심으로 문서 접근이 허용됨
- 조회/수정/삭제 권한을 역할별로 세분화하는 정책은 아직 보강 여지가 있음

## 9. MSA 전환 요구 사항 검수

### 9.1 검토 대상 요구 사항

- Spring Cloud Gateway를 이용해서 사용자 인증 절차를 공통 처리
- 요청 헤더 JWT 토큰을 해석해 `X-USER-ID` 형식으로 변경하여 마이크로서비스에 전달
- 백엔드를 마이크로서비스 단위로 구분
- 마이크로서비스 간 통신은 `FeignClient` 이용
- 마이크로서비스 간의 실시간 메시지 전송을 위한 Message Broker로 Kafka 또는 RabbitMQ 사용
- 서비스들 간 호출 URL은 `docker-compose` 서비스명 사용

### 9.2 검수 결과 요약

전체 방향은 적절하다.  
다만 그대로 구현에 들어가면 서비스 간 결합도, 내부 신뢰 모델, 운영 복잡도가 빠르게 증가할 수 있으므로 몇 가지 표현과 원칙을 먼저 보완하는 것이 좋다.

### 9.3 항목별 검수

#### 1) Gateway 공통 인증 처리

- 적절한 방향이다.
- 다만 `X-USER-ID` 하나만 전달하면 이후 권한 처리와 추적 정보가 부족할 수 있다.
- 최소한 `X-USER-ID`, `X-USER-LOGIN-ID`, `X-USER-ROLE`, `X-REQUEST-ID` 정도는 함께 고려하는 것이 좋다.
- 내부 서비스는 "Gateway가 주입한 헤더만 신뢰한다"는 원칙이 필요하다.

#### 2) 백엔드 마이크로서비스 분리

- 적절한 방향이다.
- 단, 기술 기준이 아니라 업무 경계 기준으로 분리해야 한다.
- 현재 프로젝트 기준에서는 `auth-service`, `team-service`, `erd-service`, `collaboration-service` 정도가 자연스럽다.

#### 3) 서비스 간 통신에 FeignClient 사용

- 즉시 응답이 필요한 내부 API 호출에는 적절하다.
- 하지만 모든 통신을 `FeignClient`로 고정하는 것은 권장되지 않는다.
- 조회/검증/동기 응답은 REST + Feign, 이벤트 전달은 Broker로 역할을 구분하는 것이 좋다.

#### 4) Kafka 또는 RabbitMQ 사용

- 방향은 적절하다.
- 다만 "실시간 메시지 전송"이라는 표현보다는 "서비스 간 비동기 이벤트 전달"이 더 정확하다.
- 현재 Yjs 기반 협업 실시간 동기화와는 역할이 다르므로 개념을 분리해서 설계해야 한다.

#### 5) docker-compose 서비스명 기반 호출

- 로컬 Docker Compose 개발 환경에서는 적절하다.
- 다만 이것을 시스템 전체의 고정 원칙으로 적기보다는 "로컬 개발 환경에서의 서비스 디스커버리 방식"으로 한정하는 것이 좋다.

### 9.4 먼저 반영해야 할 개선 사항

#### 1) Gateway 책임을 명확히 정의

- JWT 검증
- 공통 인증 처리
- 사용자 식별 헤더 주입
- 요청 추적 헤더 생성
- 공통 CORS 정책
- 라우팅

#### 2) 내부 헤더 신뢰 정책 수립

- 외부 요청은 반드시 Gateway를 통해서만 내부 서비스에 접근하도록 구성
- 내부 서비스는 클라이언트가 직접 보낸 `X-USER-*` 헤더를 신뢰하지 않도록 처리
- 가능하면 내부 전용 서명 또는 네트워크 격리 정책을 함께 고려

#### 3) 동기/비동기 통신 기준 분리

- 즉시 응답이 필요한 조회/검증: REST + `FeignClient`
- 상태 변경 전파, 알림, 로그, 후처리: Kafka 또는 RabbitMQ

#### 4) 서비스 경계 명확화

- 인증/사용자
- 팀/권한
- ERD 문서/메타데이터
- 협업/WebSocket

#### 5) 데이터 소유권 원칙 추가

- 서비스별 데이터 소유권 분리
- 타 서비스 DB 직접 조회 금지
- 서비스별 스키마 또는 DB 분리를 기본 방향으로 검토

### 9.5 수정 권장 요구 사항

- Spring Cloud Gateway에서 JWT 검증, 공통 인증 처리, 사용자 식별 헤더(`X-USER-ID`, `X-USER-ROLE` 등)와 요청 추적 헤더를 내부 서비스에 전달한다.
- 백엔드는 업무 경계 기준으로 마이크로서비스(`auth`, `team`, `erd`, `collaboration` 등)로 분리한다.
- 즉시 응답이 필요한 서비스 간 통신은 REST + `FeignClient`를 사용한다.
- 비동기 이벤트 전달은 Kafka 또는 RabbitMQ를 사용한다.
- 로컬 Docker Compose 환경에서는 서비스 간 호출 호스트로 `docker-compose` 서비스명을 사용한다.
- 외부 클라이언트 요청은 반드시 Gateway를 통해서만 내부 서비스에 접근하도록 구성한다.
- 서비스별 데이터 소유권을 분리하고, 타 서비스 DB를 직접 참조하지 않는다.

### 9.6 현재 프로젝트 기준 권장 판단

현재 프로젝트는 아직 단일 백엔드에서 핵심 비즈니스 흐름이 빠르게 변하는 단계이므로,  
MSA 전환은 "즉시 전체 분해"보다 "경계 정의 -> Gateway 도입 -> 서비스 분리 -> 이벤트 분리" 순서로 점진적으로 진행하는 것이 더 안전하다.

세부 전환 절차와 설계 기준은 별도 문서 `MSA_PLAN.md`를 참조한다.
